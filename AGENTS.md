## 2. 项目定位

本项目是一个面向学生简历展示、实习求职和面试讲解的 B/S 架构 Web Demo。

项目来源于“长春国际汽车城&一汽物流杯”第八届全国大学生物流设计大赛案例文件。原赛题文件提供多个可选案例，参赛队伍需要任选 2 个以上题目构成参赛主题。本项目从原比赛方案中提炼出两个核心方向：

1. 案例 14：一汽物流智慧物流园整车库内管理与优化设计；
2. 案例 16：如何实现整车售前短驳运输“零公里”。

本项目需要特别强调：

- 案例 14 中，本项目重点做“备车问题”，即待出库整车如何分配到备车道，以及在备车道内如何排序；
- 案例 16 中，原比赛方案曾使用 TruckSim 对不同路况、不同车型、不同天气下的短驳运输方案进行仿真对比，并进行方案选型；
- 当前准备用 Codex 重建的是“备车优化 + Web 端展示”部分；
- TruckSim 仿真文件目前已经找不到，因此 Codex 不需要复现 TruckSim 仿真，也不需要生成假仿真结果；
- Web 端可以保留一个“短驳仿真结果”空白/占位页面，用于说明该模块在原比赛中存在，但当前版本暂不复现；
- 不允许编造真实企业数据、真实 TruckSim 输出、真实上线结果或真实业务收益。

本项目的核心目标是：

> 把原比赛中的“备车遗传算法 + Web 客户端呈现”重构为一个可运行、可解释、可展示的简历项目。

## 4. 业务背景说明

### 4.1 案例 14：备车问题

一汽物流智慧物流园是整车自主机厂下线后的第一站。整车进入物流园后，仓储管理系统 VWMS 会根据车型、车龄、颜色、目的地等信息，将入库整车匹配到园区内库位。

在整车出库阶段，系统需要根据 VWMS 排布的出库时间，按照多种业务约束，将待出库整车安排到各个备车道，并给出每条备车道内的排序。

案例 14 中与本项目相关的业务约束包括：

1. 按照先进先出原则安排；
2. 目的地相同或同方向的车辆尽量分到同一组；
3. 车型大小需要与公路轿运货车装载条件匹配；
4. 需要考虑公路轿运货车备车顺序；
5. 通常以 8 辆整车为一组指定备车道；
6. 整车进入备车道后，需要在 1 小时内完成装载并驶离园区；
7. 备车道分配和排序会影响后续装车效率，以及是否发生多次装卸。

因此，本项目将“备车问题”抽象为：

> 在给定待出库整车集合、目的地、方向、车型、颜色、出库时间、优先级和备车道容量的情况下，生成备车道分配与排序方案，使同方向/同目的地车辆尽量集中，尽量满足先进先出和装载匹配规则，并提升备车道利用率和装载效率。

### 4.2 案例 16：短驳问题

案例 16 关注整车售前短驳运输“零公里”问题。整车物流逐渐从公路直送为主转向多式联运，短驳运输成为影响成本、效率、安全和售前行驶里程的重要环节。

原比赛方案中，短驳部分曾使用 TruckSim 对以下场景做仿真对比：

1. 不同路况；
2. 不同车型；
3. 不同天气；
4. 不同短驳运输方式。

并基于仿真结果进行方案选型和评价。

但当前重建项目时，TruckSim 仿真文件已经缺失。因此，Codex 不需要实现短驳仿真，也不需要展示具体仿真结果。Web 端仅保留一个“短驳仿真结果”页面，页面中说明：

> 原比赛方案中曾基于 TruckSim 进行短驳运输仿真对比，用于不同路况、车型、天气下的方案选型。当前开源/简历展示版本暂不包含 TruckSim 原始仿真文件，因此本模块仅保留页面入口和方法说明，不展示具体仿真数据。

## 6. 备车算法设计

### 6.1 问题抽象

输入：

- 待出库整车集合；
- 备车道集合；
- 每条备车道容量；
- 每辆车的车型、颜色、目的地、方向、下线时间、出库时间和优先级。

输出：

- 每辆车被分配到哪条备车道；
- 每条备车道内车辆的排序；
- 各项评价指标。

目标：

1. 同目的地车辆尽量集中；
2. 同方向车辆尽量集中；
3. 尽量满足先进先出；
4. 尽量提高备车道利用率；
5. 尽量减少未分配车辆；
6. 尽量减少车型装载冲突；
7. 保持每条备车道不超过 8 辆车。

### 6.2 基线规则算法

先实现一个可解释的规则算法作为对照：

1. 按 `outbound_time` 从早到晚排序；
2. 再按 `destination` 和 `direction` 分组；
3. 每组按 8 辆切分；
4. 依次分配到空闲备车道；
5. 输出分配结果。

该算法用于和遗传算法对比。

### 6.3 遗传算法编码

建议使用排列编码或分组编码。

第一版建议使用简单可实现的排列编码：

```text
染色体 = 所有 car_id 的一个排列
```

解码方式：

1. 按染色体顺序读取车辆；
2. 每 8 辆形成一个备车组；
3. 每个备车组依次分配到备车道；
4. 组内顺序即为备车道内排序。

### 6.4 适应度函数

适应度函数需要综合多个目标。第一版可以设计为惩罚项越小越好。

建议目标函数：

```text
fitness = 
    w1 * destination_penalty
  + w2 * direction_penalty
  + w3 * fifo_penalty
  + w4 * utilization_penalty
  + w5 * unassigned_penalty
  + w6 * model_mix_penalty
```

其中：

#### destination_penalty

同一备车道内目的地越分散，惩罚越高。

#### direction_penalty

同一备车道内方向越分散，惩罚越高。

#### fifo_penalty

如果下线时间更早或出库时间更早的车辆被排在后面，产生 FIFO 违背惩罚。

#### utilization_penalty

备车道未装满时产生惩罚。

示例：

```text
utilization_penalty = sum(abs(8 - lane_vehicle_count))
```

#### unassigned_penalty

无法分配到备车道的车辆产生较高惩罚。

#### model_mix_penalty

如果同一备车道内车型组合明显不利于装载，可以设置惩罚。第一版可以简化处理，例如车型过于分散时增加惩罚。

### 6.5 遗传算法流程

1. 生成初始种群；
2. 解码染色体；
3. 计算适应度；
4. 选择；
5. 交叉；
6. 变异；
7. 保留最优个体；
8. 重复迭代；
9. 输出最优备车方案；
10. 输出收敛曲线。

第一版参数建议：

```text
population_size = 50
generations = 100
crossover_rate = 0.8
mutation_rate = 0.1
elite_size = 2
lane_capacity = 8
```

## 7. 后端 API 设计

后端使用 FastAPI。

建议接口：

```text
GET  /api/health
GET  /api/cars
GET  /api/staging-lanes
POST /api/staging/baseline
POST /api/staging/genetic
GET  /api/staging/comparison
GET  /api/project-info
```

### 7.1 /api/staging/baseline

运行基线规则算法。

返回：

- 备车道分配结果；
- 指标统计；
- 算法说明。

### 7.2 /api/staging/genetic

运行遗传算法。

请求参数可以包括：

```json
{
  "population_size": 50,
  "generations": 100,
  "crossover_rate": 0.8,
  "mutation_rate": 0.1
}
```

返回：

- 最优备车道分配结果；
- 每条备车道内排序；
- 指标统计；
- 最优适应度；
- 收敛曲线；
- 参数配置。

### 7.3 /api/staging/comparison

返回基线算法与遗传算法对比。

对比指标：

- 使用备车道数量；
- 平均利用率；
- 同目的地集中度；
- 同方向集中度；
- FIFO 违背次数；
- 未分配车辆数；
- 综合惩罚值。

> 这个项目来源于全国大学生物流设计大赛的一汽物流案例。原赛题包含多个方向，我们团队当时节选了案例 14 和案例 16 相关内容形成参赛主题。其中案例 14 我主要聚焦备车问题，也就是整车出库前如何按照目的地、方向、先进先出和轿运车装载规则，将待出库车辆分配到备车道并确定排序。
>
> 备车问题本质上是一个带业务约束的组合优化问题。我在项目中将车辆序列作为遗传算法染色体，通过解码得到不同备车道的车辆分组与排序，再基于同目的地集中度、同方向集中度、FIFO 违背次数、备车道利用率和未分配车辆数设计适应度函数。为了便于对比，我还实现了一个按出库时间和目的地方向分组的基线规则算法。
>
> 案例 16 是短驳问题，我们当时使用 TruckSim 做过不同路况、不同车型、不同天气下的仿真对比，用于方案选型。但由于原始仿真文件目前找不到，所以这次 Web 重构没有编造仿真结果，而是在前端保留了短驳仿真占位页面，说明该模块的研究方法和后续扩展方向。
>
> 最终我用 FastAPI 和 React 搭建了一个 B/S 架构 Demo，重点展示备车优化输入、算法运行、结果分配、指标对比和收敛曲线。这个项目体现了我将物流业务问题抽象为优化模型，并通过 Web 系统呈现算法结果的能力。

## 15. 禁止事项

1. 不要声称当前系统已经接入一汽物流真实业务；
2. 不要声称当前系统真实上线；
3. 不要使用真实企业内部数据；
4. 不要编造 TruckSim 仿真文件；
5. 不要编造短驳仿真结果；
6. 不要把模拟数据说成真实数据；
7. 不要过度开发非核心功能；
8. 不要让短驳页面喧宾夺主；
9. 不要让 Codex 把重点转移到地图、无人驾驶或 TruckSim 复现；
10. 当前最重要的是完成“备车遗传算法 + Web 端可视化”。
